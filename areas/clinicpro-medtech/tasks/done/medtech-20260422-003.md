All three tables in `public` schema have RLS disabled. The anon key (`NEXT_PUBLIC_SUPABASE_ANON_KEY`) is in the browser bundle, meaning anyone with it can query these tables directly via PostgREST without going through the API routes.

Tables are server-side only (accessed via service role key, which bypasses RLS), so functionally correct. But enabling RLS with no policies blocks anon/authenticated role access as a defence-in-depth measure.

Run in Supabase dashboard SQL editor:

```sql
ALTER TABLE public.medtech_image_commit_audit ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.rate_limit_buckets ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.users ENABLE ROW LEVEL SECURITY;
```

No policies needed — service role bypasses RLS. Verify app still works after applying (audit log writes use service role via Drizzle, so no breakage expected).
