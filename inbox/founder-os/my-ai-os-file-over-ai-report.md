## Overview

Nick Milo (Linking Your Thinking) presents a framework for building a personal "AI Operating System" (AI OS) grounded in locally owned plain-text files. The argument is aimed at knowledge workers who want their AI workflows to survive tool churn, model deprecation, and company pivots. The practical output is three Markdown files (`me.md`, `vault-map.md`, `skill-map.md`) that act as a portable identity and navigation layer for any AI tool.

---

## Core Thesis

Users are currently building on rented infrastructure. When a tool changes, an API breaks, or a company pivots, they lose workflows and potentially data. The solution is to anchor everything to locally owned Markdown files (the "Ideaverse") and separate that stable layer from the volatile layer of AI tools. Milo calls this "File over AI," an extension of Obsidian CEO Kepano's "File over app" philosophy. The three Maps & Manuals files are the connective tissue: they carry your identity, your vault structure, and your defined AI skills into any AI tool without lock-in.

## Transcript Summary

**0:00 - 1:29 | Problem and philosophy**
- AI tool proliferation creates lock-in risk: "when the API breaks, the company pivots, or your favourite model gets deprecated," you lose ideas.
- Obsidian stores notes as plain `.md` files, readable in any text editor (TextEdit, VS Code, iA Writer all shown).
- Kepano's "File over app" principle: "Apps are ephemeral, but your files have a chance to last." Milo extends this to AI.
- Claude service disruption error message shown as concrete evidence of cloud tool unreliability.

**2:08 - 5:10 | Three-layer AI OS model**
- **Layer 1: Ideaverse** -- personal knowledge universe (notes, ideas, connections). Organised as ACE: Atlas (reference knowledge), Calendar (time-based notes), Efforts (projects). This is the stable core.
- **Layer 2: Maps & Manuals** -- a small set of Markdown files that sit between the Ideaverse and any AI tool. They make large vaults (e.g., 20,000+ notes) navigable by AI without burning tokens on brute-force search. Described as "connective tissue" and "indexes."
- **Layer 3: Tools** -- the actual AI applications (Claude, NotebookLM, OpenAI, Gemini, etc.). Explicitly called "the most replaceable layer." News headlines about model deprecations and app shutdowns reinforce this.

**5:10 - 12:29 | The three core files**

`me.md` (Portable Identity):
- A Markdown file that acts as a universal briefing for any AI: who you are, how you think, how you want AI to work with you.
- Contains: Summary statement, first principles, values, working preferences, key frameworks, intellectual lineage.
- Unlike `CLAUDE.md` (tool-specific), `me.md` is universal and "rarely changes."
- Strategy shown: `CLAUDE.md` simply redirects: "Go immediately to /me.md and you will find everything there."

`vault-map.md` (Ideaverse Navigation Manual):
- AI's instruction manual for the vault structure: how folders are organised, how to create notes (templates, naming conventions), how to interact with other Ideaverses (team, family).
- Changes more often than `me.md` (when priorities or vault structure shift).
- Also describes multi-Ideaverse interaction (LYT Team Ideaverse, Milo Family Ideaverse shown as examples).

`skill-map.md` (Skills Index):
- A Markdown index of reusable AI process files (e.g., `holistic-briefing`, `daily-note`, `summarize-article`).
- Individual skill files contain structured steps and output format for AI to follow.
- Organised under categories: Briefings, Email, Files & Maintenance, Meetings.
- Skills are portable across AI tools.

**12:29 - 14:20 | Privacy and next steps**
- Using frontier AI models means trusting Layer 3 tools with some data. Milo acknowledges this trade-off.
- "Playing Defence" rules: (1) Keep personal content outside AI zones. (2) Make backups. (3) Mark AI-generated content clearly.
- Long-term belief: open source and local models will eventually be capable enough for most personal use cases.
- Promotes "Linking Your AI" course and a follow-up video on using the AI OS with Obsidian and Claude.

## Key Takeaways

- Plain-text files are the only durable substrate for AI workflows. Any system built solely inside a proprietary AI tool is at risk.
- The three-layer model (Ideaverse / Maps / Tools) separates stable personal assets from volatile external tools.
- `me.md` is the most portable file: a universal AI briefing that moves across tools unchanged.
- `vault-map.md` solves the token-burn problem for large vaults: AI navigates via the map rather than scanning thousands of files.
- `skill-map.md` makes repeatable processes portable and explicit. Skills are Markdown files, not stored in any one tool's memory.
- Tool-specific files (e.g., `CLAUDE.md`) should redirect to `me.md` rather than duplicate identity content.
- The Tools layer is explicitly the most replaceable. Design for tool death from the start.
- Multi-Ideaverse interaction (team, family) is possible in principle; practical implementation not demonstrated.
- "Playing Defence" requires active decisions: what goes into AI zones, what stays out, and how AI-generated content is labelled.
- The long-term bet is local/open-source AI becoming capable enough for personal use, removing cloud trust requirements entirely.

## Steps Covered

1. Create ACE folder structure: `Ideaverse/Atlas`, `Ideaverse/Calendar`, `Ideaverse/Efforts`.
2. Create `AI OS/` folder inside `Ideaverse/`.
3. Create `me.md` inside `AI OS/` with: Summary, First Principles, Working Preferences, Frameworks, Intellectual Lineage.
4. Create a tool-specific redirect file (e.g., `CLAUDE.md`) pointing to `me.md`.
5. Create `vault-map.md` documenting folder structure, note creation rules, and inter-Ideaverse interaction.
6. Create `skill-map.md` as an index of skill files.
7. Create individual skill files (e.g., `holistic-briefing.md`) with structured steps and output formats, stored in `AI OS/Skills/`.
