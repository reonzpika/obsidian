## Context

Raised by Lawrence Peterson in the 22 April meeting. He wants a management dashboard so Medtech AU can manage practices and licences. Described as a tenancy model: "buy more licences as needed." This is out of scope for the current AU bundle agreement (scope is locked to image capture + ALEX DocumentReference POST). Will be invoiced as a separate one-off build engagement.

Cannot price or commit until scope is known. Scope must come from Lawrence.

## What we know

- Medtech needs to onboard and manage AU practices centrally
- Tenancy model: Medtech adds licences as more practices join
- Likely Medtech-facing admin portal (not per-practice)
- Auth model for this admin layer is unknown

## Ideation — what this dashboard probably needs

### Core features (likely required)
- Practice onboarding: add/remove practices, assign users per practice
- Licence management: view current count, expand (trigger for ClinicPro to invoice)
- User management: add/remove users at practice level from Medtech side
- Usage visibility: active practice count, capture activity per practice

### Possible additions (needs Lawrence confirmation)
- Audit log view per practice
- Billing/invoice history
- Support tools: reset access, view errors per practice

### What it is NOT (hold the line)
- Not a GP-facing interface (that is the existing Capture product)
- Not automated billing/Stripe integration (invoicing remains manual for now)

## Build approach (preliminary)

With AI assistance, a focused admin portal is tractable. Likely:
- Next.js admin route group, separate from the clinical app
- Supabase RLS with a Medtech admin role above practice level
- Same codebase as Capture (admin module), not a new repo
- Auth: either Medtech SSO (if they want it) or ClinicPro-issued admin credentials

## Next steps

1. Get scope requirements from Lawrence before pricing or building
2. Confirm it is out of scope for the bundle agreement, separate invoice
3. Once scope is clear: estimate build effort and draft one-off invoice
4. Clarify auth model for the admin layer (ties into the unresolved auth question for the main product)
